    Easy wins with Cython

    Cython

    Cython is a compiler that allows for extensions to Python written in C. This overcomes two of the biggest drawbacks of Python for data analysis: the global interpreter lock (GIL) and its dynamic typing.

    Using Cython one can use static typing to speed up core functions and loops, making huge speed gains even when compared to well vectorised code. Furthermore, by removing the GIL for pure C sections of code one can use multiple cores using OpenMP.

    Let's go through some examples.

    import cython
%load_ext Cython

    The Cython extension is already loaded. To reload it, use:
  %reload_ext Cython

    def summer(n):
    S = 0
    for i in xrange(n):
        for j in xrange(n):
            for k in xrange(n):
                S += i ** 2 + j ** 2 + k ** 2
    return S

    N = 100
%timeit summer(N)

    1 loop, best of 3: 219 ms per loop

    Of course, this is not how we should write Python, for loops are notoriously slow in Python due to the large overhead of typing at run-time. However, cases in which nested loops are unavoidable are the cases in which Cython makes the easiest and most significant gains.

    Let's now write the same function again, but in Cython. To do this we need to declare the variable types using the cdef keyword, exactly as we would do if we were writing normal C. Note that we are using the %%cython magic command - this is because Cython is actually a compiler, and what is going on behing the scenes is that out code is being compiled by Cython, which understands how to implement the cdef typing. 

    %%cython -a
def cython_summer(int n):
    cdef int S
    cdef int i
    cdef int j
    cdef int k
    for i in range(n):
        for j in xrange(n):
            for k in xrange(n):
                S += i ** 2 + j ** 2 + k ** 2
    return S

    By giving %%cython the -a flag we get an annotated output of the code analysed by Cython, and the C code it generates. The more yellow a line, the more Python overhead it contains. This can be useful for identifying the areas of code with a large Python overhead, these areas will benefit most from being Cythonized. Let's now time this:

    %timeit cython_summer(N)

    That's almost a 3 order of magnitude speed up compared to the previous 212ms! But we can go even further! By writing the expensive part of the function in pure C, we are allowed by Python to disable the GIL and run the computation on multiple CPUs. This requires there to be no references to Python objects in the region where GIL is disabled. Doing this will cause you nightmares.

    We must also pass Cython extra flags to tell it to use openmp (these will depend on your platform, for Linux use -fopenmp). We also need to use the cython.parallel.parallel() method to specify the number of threads to use, and the cython.parallel.prange() method to tell OpenMP to distribute these loops between threads (i.e. super-easy to make parallel gains for 'embarrassingly parallel' code). Giving guided to the keyword argument schedule means that the threads will be awarded jobs as they become availale.

    %%cython --compile-args=-fopenmp --link-args=-fopenmp --force -a
from cython.parallel import parallel, prange, threadid
def cython_parallel_summer(int n):
    cdef int S
    cdef int i
    cdef int j
    cdef int k
    with nogil, parallel(num_threads=2):
        for i in prange(n, schedule='guided'):
            for j in range(n):
                for k in range(n):
                    S += i ** 2 + j ** 2 + k ** 2
    return S
